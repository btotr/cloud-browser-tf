A cloud browser is a browser running and execution on a server. This document describes the concepts and architecture for the Cloud Browser. The main purpose is to provide the building blocks for a cloud browser solution.

This is a public working draft that the Cloud Browser Task Force is discussing and exploring. It has no official standing of any kind and does not represent the support or consensus of any standards organisation or contributor. This is a subset of the W3C Public Web & TV Interest Group - Cloud Browser Task Force. As the Cloud Browser TF progresses its work, this section will be used to identify the architecture that has reached rough consensus within the group.

Introduction

Today's browsers need a vast amount of hardware resources to process modern web sites or simply unable to support modern web technologies such as HTML5. The Cloud Browser concept addresses these issues by putting the browser into a more powerful, easier and manageable server or Cloud. The service execution is shifted to the Cloud, where the user interface is rendered and streamed down as a media stream to the client. The main functionality of the client is decoding and presenting the media stream to the end user. Using this design, it is possible to provide a uniform user interface for a large range of devices. Furthermore, this concept reduces the need for processing power of the client and helps in deploying new browser technologies faster.

Architecture

The following section provides an overview of the main building blocks of the cloud browser solution. The goal would be to have a consistent architecture between cloud vendors but doesn't limit the inovation of specific implementations.

client device

A client device is the actual hardware where the User Interface will presented. There are no requirements to the client device other than it should have a runtime environment. a client device could be anything from a set-top-box to a smarttv or even mobile phone.

Runtime enviroment

In a cloud browser architecture the execution is done on a server and the result is send down to a so-called runtime environment. This runtime environment is only responsible for displaying the stream and provide essential information such-as key presses.The latter is depending on the infrastructure. In the near future there may be a specification how to exchange this information for example with json-dl with a generic communication mechanisms such as websockets or webRTC. The transport and format will be agnostic. i.e. vendor specific but may be standardised outside w3c. Furthermore the runtime environment doesn't have any context. i.e. it will connect to the orchestration and displays the result. This will make sure that all the logic is within the cloud and no necessary updates must be made on the client device. This is also a fundamental different with a transforming proxy where the context still exist on the client device. An easy way to visualise the run time environment is to see it as a remote display which also act as gateway to send essential information to the cloud browser.

Cloud browser

The cloud browser it self is a browser instance on the server and will act as any other conventional browser.

Orchistration

The cloud browser lives in an so-called orchestration which is mainly responsible for session management and abstraction. It is also responsible how the stream will send to the runtime environment as this is depending on the underlaying infrastructure. There are two main approaches, and double stream which are describe below.

Single Stream

The maximum number of functionalities that can be shifted into the Cloud environment is referred to as a single stream approach of a Cloud Browser.

Double Stream

With regards to the double stream approach of a Cloud Browser, the Cloud Browser renders the user interface only, while the media is delivered from another server. Thus, the user interface and media streams are delivered separately to the client, which then has to combine both of these streams and present them to the end user in a unified form. This approach therefore enables reuse of the legacy video delivery infrastructures such as multicast networks.
